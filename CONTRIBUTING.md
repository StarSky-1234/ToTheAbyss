# ToTheAbyss Project Contribution Specification
## 概述
本文档规定本项目代码开发、议题管理、版本提交、分支协作的统一标准，所有参与项目开发人员均需遵照执行。
本规范基于通用前端开发行业惯例制定，不存在违背常规开发习惯的强制约束；若存在特殊业务场景无法遵循本规范，可在对应代码位置添加注释，说明特殊处理原因。

## 一、代码编写规范
### 1. 代码格式规范
1. 统一使用2个空格作为代码缩进，禁止同一文件内混用Tab制表符与空格缩进；
2. 运算符、赋值符号两侧保留单个空格，提升代码可读性；
3. 功能代码块前后空行分隔，用于区分不同逻辑段落；
4. 正式提交至仓库的代码，需移除临时调试打印语句`console.log`；若业务场景需要保留日志输出，须统一调用项目内置日志工具函数。

### 2. 注释规范
1. 业务复杂逻辑、特殊边界处理、已知缺陷点位，必须添加注释说明；
2. 简单通用、逻辑直观的基础代码无需添加冗余注释，避免代码冗余；
3. 待优化、待修复代码统一使用标准标记：
   - `// TODO: 待优化内容 关联Issue编号`：标记后续迭代优化任务
   - `// FIXME: 缺陷描述`：标记当前代码存在的程序问题

### 3. 样式文件规范
CSS 及相关样式文件按「布局定位 → 盒模型 → 文本样式 → 特殊修饰」顺序分组编写，同类样式集中放置。

## 二、资源与代码命名规范
### 1. 文件、目录命名
1. 业务目录、静态资源文件夹采用短横线命名法（kebab-case），示例：`map-config`；
2. 组件类 JS/TS 文件采用大驼峰命名（PascalCase），示例：`MapLoader.ts`；普通工具脚本采用小驼峰命名法；
3. 图片、静态资源文件统一使用小写英文命名，多词汇间以下划线分隔。

### 2. 代码内标识符命名
1. 普通变量、函数：小驼峰 camelCase，示例：`mapLoadStatus`、`getTerrainData()`；
2. 全局固定常量：全大写+下划线分隔，示例：`MAX_MAP_SIZE`；
3. 布尔变量推荐使用`is/has/can`前缀，不作强制硬性要求；
4. 类、构造函数：大驼峰 PascalCase；CSS 类名：短横线命名。

### 3. 神奇代码岛地图命名规范
1. 地图基础命名格式：「前缀」+【天梯S3】To The Abyss 直抵深渊
2. 前缀定义及用途：
- `copy`：地图备份版本，仅用于本地留存；
- `wasted`：废弃地图，禁止用于正式开发，项目发布前需全部删除；
- `main`：主地图，对应正式发布可用版本。
3. 项目正式发布前，需移除地图名称内的前缀，保证对外展示名称统一为标准名称。

## 三、Git 版本管理规范
### 1. 分支管理规则
1. `main` 为项目稳定主分支，仅存放可正常运行、完成自测的完整版本，禁止直接推送未验证的开发代码至主分支；
2. 功能开发分支命名标准：
   - 缺陷修复分支：`fix/问题简要描述`
   - 新功能开发分支：`feature/功能名称`
3. 小型文本、配置文件等低风险改动，可直接在 `main` 分支提交，无需新建独立开发分支。

### 2. 提交信息规范
采用约定式提交（Conventional Commits）标准，提交消息以固定类型前缀开头：
- `fix`：修复程序缺陷、BUG
- `feat`：新增业务功能
- `docs`：修改文档、开发规范、README 等文本文件
- `refactor`：代码重构，不改变程序外部运行逻辑
- `chore`：依赖更新、构建配置、工具脚本调整

标准提交示例：
`fix: 修复地图加载界面卡死问题 #1`
提交描述需清晰说明本次改动内容，无强制字数限制。

## 四、Issue 项目管理规范
1. 新建 Issue 必须选用仓库内置模板：Bug report、Feature request、Task，并按模板完整填写全部信息；
2. 根据议题类型、优先级绑定对应标签：`bug`/`enhancement`/`documentation`/`urgent` 等；阻塞整体开发进度的任务需标记 `urgent`；
3. 所有 Issue 统一录入项目看板，并绑定对应开发里程碑，完整跟踪任务全生命周期；
4. Bug 类 Issue 需完整填写复现步骤、运行环境，附带报错截图、运行日志，便于快速定位故障根源。
